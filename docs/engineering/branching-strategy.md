# Branching Strategy

- `main` должен оставаться в собираемом состоянии
- новые изменения вносить через короткие ветки `codex/...`
- работа ведется по story: одна story = одна рабочая ветка = один PR
- merge в `main` делается только после review и зеленых checks
- squash merge предпочтителен, если не нужен подробный commit trail
- архитектурные изменения должны сопровождаться ADR или обновлением docs
